## What Went Well

- **Code-review skill caught the deliberate bug** — The off-by-one (`1023` instead of `1024` in the boundary test) was found independently by both the LLM review phase and Gemini Flash, with no hints. Specific `file:line` references were correct. The skill held up under adversarial conditions.
- **Checking MDX docs before implementing** — After the user pushed back on the hand-rolled `declare module '*.mdx'`, fetching the official MDX TypeScript docs led to the correct solution (`@types/mdx` + `mdx/types.js` augmentation) rather than guessing.
- **Gemini fallback chain worked** — When `gemini-2.5-pro` hit capacity limits, falling back to `gemini-2.5-flash` via `env -u` project var reset succeeded and returned useful output both times it was tried.
- **Commit messages captured cross-cutting context** — The test infrastructure commit noted that the work was done while testing the code-review skill, which preserves the reasoning in git history without needing a comment in the code.
- **TypeScript upgrade surfaced and fixed a pre-existing issue** — Upgrading TS 5.1.6 → 5.9.3 revealed the missing MDX type declarations. Rather than patching around it, the root cause was fixed properly.

## What Went Wrong

- **checks.sh was over-engineered on first attempt** — The initial script had argument parsing, tool detection with env resets, a dependency audit phase, and multi-format output. The user rejected it immediately as too complicated. A rewrite was needed.
- **Hand-rolled MDX type declaration written before checking docs** — Created `declare module '*.mdx'` from memory rather than looking up the official approach first. User had to intervene and point to the docs.
- **vite-tsconfig-paths installed then immediately uninstalled** — Installed as part of the original plan, removed one step later when Vite's native `resolve.tsconfigPaths` was discovered. A wasted install/uninstall cycle.
- **Plan mode exit flow was bumpy** — Tried to call `ExitPlanMode` before saving the plan to the user's preferred location (`.claude/plans/plan-write-tests.md`), causing the exit to be rejected and requiring a manual save first.

## Time Wasted

- **checks.sh rewrite** — ~5 min. Wrote a complex script that didn't match the user's intent ("just run the tools and output results").
- **MDX type declaration rework** — ~5 min. Writing the wrong solution first, then re-doing it against the docs.
- **vite-tsconfig-paths install/uninstall** — ~2 min. Wasted npm round-trip.
- **Gemini capacity retries** — ~10 min of background wait across two calls that hit `gemini-2.5-pro` exhaustion before falling back to flash.

## Root Causes

- **checks.sh over-engineering**: No spec for complexity level — I defaulted to "robust and general-purpose" when the user wanted "minimal and readable." The skill's own description says "run the tools and output structured results" — that should have been sufficient guidance.
- **MDX types from memory**: Writing code for unfamiliar APIs without checking docs first is the failure mode the user specifically guarded against by pushing back. The pattern to follow is: if it touches a third-party type system, check docs first.
- **vite-tsconfig-paths**: Didn't check whether Vite had native support before installing a plugin to do the same job. A quick `vitest --help` or checking the Vite changelog would have caught this.
- **Plan save location**: The `ExitPlanMode` call happened before confirming the plan was where the user expected it. Should save to the project-local path first, then exit.
